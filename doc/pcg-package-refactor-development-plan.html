Meshova「稳定内核 + PCG 包」重构开发计划

只做架构与迁移计划，不修改现有源码。面向普通 TypeScript、Web 实时预览、URL 传播、AI 阅读与扩展。

一句话方案

保留普通 TypeScript 脚本；取消节点蓝图路线。内核只提供稳定、领域无关的算法与运行契约。所有模型、材质、领域生成逻辑、参数定义、展示元数据进入可版本化 PCG 包。

首期不拆多个 npm 产品：仓库内先落成 一个稳定内核 + 一个官方 PCG 包。领域分类只是包内目录。以后确有独立发布需求，再拆包。

硬指标：新增模型或材质时，核心源码零修改、Web 查看器零手写注册、普通 TS 可直接阅读和调试。

1. 核心决策

问题决策原因
节点蓝图不做不保存图、节点 ID、端口、连线。避免两套作者系统与 AI 阅读噪声。
内容事实源普通 TypeScript模型与材质保持函数、条件、循环、抽象、重构能力；可读、可测、可 Diff。
系统分层稳定内核 + PCG 包只保留两层发布概念。领域库与内容配方使用同一 PCG 包机制。
首期包数量一个官方 PCG 包减少目录、版本、依赖、发布复杂度。按领域分目录，不急于拆发布单元。
领域逻辑放 PCG 包内车辆、建筑、植被、道路、织物等逻辑仍会变化，不承诺核心级稳定性。
注册方式构建期自动目录新增内容不再改 web/procmodels.js、web/materials.js 或根导出表。
现有 OpPlan / WorkflowPreset停止作为新内容标准它们仍是图式数据。保留兼容或内部诊断价值，不作为 PCG 包作者格式。
明确排除：不把 TS 编译成可编辑节点；不从节点反写脚本；不把模型保存成 OpPlan JSON；不要求领域函数进入核心后才能被 Web 使用。

2. 现状与根因

2.1 当前混合方式

位置当前职责问题
src/models/具体模型、默认参数、领域辅助函数、展示属性随根库统一导出；增加内容等同扩张内核。
src/texture/纹理缓冲/滤镜/PBR 基础 + 大量具体材质与批次内容底层算法与内容配方混放；稳定性边界不清。
src/geometry/网格基础 + 道路、河流、岩石、建筑等领域算法通用算法与高频变化领域系统混放。
src/vegetation/曲线框架、枝叶、生长、物种、编辑、LOD既含通用算法，也含完整领域产品逻辑。
web/procmodels.js模型参数 Schema、包装 build、分类、空间编辑配置10,844 行手工注册；模型实现与 Web 接入重复描述。
web/materials.js多个材质表拼接、标签、分类、运行适配每批材质需要扩表；浏览器目录成为第二事实源。
examples/模型/材质入口、导出脚本、演示大量一对一包装；与内容目录重复。
src/index.ts导出几乎全部模块，包括 models公共 API 面积随内容增长；无法稳定版本语义。

2.2 根因

  • “能复用”被等同于“应进入核心”。领域函数复用几次后直接进入 src 公共面。
  • 模型脚本只保留参数和调用；真正造型逻辑在内核模块。新模型缺能力时只能继续改核心。
  • 内容缺少统一自描述入口。Web、CLI、测试、示例各自维护注册信息。
  • 基础算法、领域工具、最终内容没有稳定性等级与晋升门槛。
  • 现有 WorkflowPreset 尝试解决参数与绑定，但以 OpPlan 图为主体，不符合“普通脚本唯一事实源”。

3. 目标与非目标

必须实现
  • 内核 API 小、领域无关、按 SemVer 稳定。
  • 模型与材质由普通 TS 完整表达。
  • PCG 包可含领域工具、模型、材质、资源、测试。
  • 新增内容不改内核、不改 Web 手工注册。
  • Node 与浏览器加载同一内容入口。
  • Seed、参数、版本可组成稳定分享链接。
  • AI 可只读单个入口及其局部依赖。
  • 现有全部模型、材质完成迁移与回归。
本轮不做
  • 节点编辑器、图序列化、脚本转蓝图。
  • 任意远程代码默认直跑主线程。
  • 一次性大爆炸搬迁。
  • 迁移期间顺手重写模型外观。
  • 为了拆包先搭复杂插件市场。
  • 把每个领域拆成独立 npm 包。
  • 把所有领域辅助函数晋升为核心 API。

4.1 内核边界

稳定留在内核迁入 PCG 包判定规则
向量、矩阵、标量、确定性随机、Noise 基础具体风格 Noise 组合、某材质专用 Pattern是否独立于资产类别与艺术目标。
Mesh / Curve / PointCloud / Field / TextureBuffer 数据结构车辆、屋顶、道路、河流、岩层、树种等组合结构是否是所有 PCG 包共同需要的数据契约。
图元、变换、合并、查询、拓扑、布尔、细分、UV、基础散布城市路网、建筑立面、生态规则、植被生长、机械装配是否可用短名称说明数学操作，而非业务名词。
纹理采样、滤镜、SDF、通道混合、PBR 数据与校验木材、织物、锈蚀、墙面、科幻面板等完整材质系统是否输出通用计算能力，而非可直接展示的成品风格。
执行上下文、缓存协议、沙箱、导入导出、错误模型模型默认参数、标签、颜色、来源元数据、缩略图是否决定“怎么运行”，而不是“生成什么”。
PCG 包契约与目录加载器官方内容目录、分类、模型/材质列表机制留核心，清单由内容生成。

核心晋升门槛

PCG 包中的函数只有同时满足以下条件，才可提案进入内核：

  • 领域无关；名称不含具体资产或风格。
  • 至少被两个独立领域真实复用，不以“未来可能复用”为依据。
  • 输入输出使用稳定核心类型；无展示元数据、资源路径、UI 假设。
  • 确定性、边界行为、错误语义已有完整测试。
  • API 已在 PCG 包中演化过，能承诺 SemVer 与弃用周期。

4.3 脚本作者格式

defineModel 与 defineMaterial 只做类型检查、元数据收集、参数校验。build 内仍是普通 TS，不形成图。

import {
  box, cylinder, merge, transform, vec3,
  defineModel,
} from "@meshova/core";
import { placeAtCorners } from "../../shared/architecture/place-at-corners.js";

interface ChairParams {
  width: number;
  depth: number;
  height: number;
}

export default defineModel<ChairParams>({
  id: "chair",
  version: 1,
  label: "四腿椅",
  params: {
    width:  { kind: "number", default: 0.62, min: 0.4, max: 1.0 },
    depth:  { kind: "number", default: 0.58, min: 0.4, max: 0.9 },
    height: { kind: "number", default: 0.82, min: 0.6, max: 1.2 },
  },
  build(params) {
    const seat = box(params.width, 0.1, params.depth);
    const leg = cylinder(0.035, params.height - 0.1, 12);
    const legs = placeAtCorners(leg, seat);
    return [{ name: "座面", mesh: seat }, { name: "椅腿", mesh: merge(...legs) }];
  },
});
import { defineMaterial, materialFromFields } from "@meshova/core";
import { brushedWoodFields } from "../../shared/materials/wood.js";

export default defineMaterial({
  id: "warm-oak",
  version: 1,
  label: "暖色橡木",
  params: {
    grainScale: { kind: "number", default: 9, min: 2, max: 24 },
    seed: { kind: "seed", default: 7 },
  },
  build(params) {
    return materialFromFields(brushedWoodFields(params));
  },
});

作者规则

  • 脚本入口短：元数据、参数、build、输出。复杂逻辑拆到同目录或 shared。
  • 入口不得依赖 Web DOM、Node 文件系统、当前时间、隐式全局状态。
  • 随机必须来自显式 Seed；包内 helper 同样遵守。
  • 参数使用语义名称；UI 不显示 root.0、component_1 等原始对象名。
  • 来源只写内部元数据；公开名称只描述内容与用途。
  • 模型与材质可互相引用稳定 ID，但禁止循环包依赖。
  • 性能关键、真正通用的算法才能进入核心；先在 PCG 包验证。

5. 加载、Web 预览与传播

5.1 自动目录

浏览器不能运行时扫描仓库。采用构建期扫描 + 生成目录 + 运行时动态导入：

  1. 扫描 *.model.ts、*.material.ts。
  2. 静态加载定义，校验 ID、版本、参数、标签、依赖。
  3. 生成 generated/pcg-catalog.ts，包含轻量元数据与动态 import。
  4. Web、CLI、Agent 统一消费生成目录。
  5. 生成文件不手改；CI 检查生成结果是否最新。

5.2 Web 查看器

  • 模型/材质列表、分类、搜索、参数面板、空间绑定全部读定义元数据。
  • 选择条目后动态加载对应脚本；不一次打包全部内容。
  • 查看器只负责通用交互，不含某模型特判。
  • 空间编辑能力由可选 bindings 元数据声明，build 接收普通类型数据。
  • 材质预览继续使用程序化实时生成，不把预览位图当材质源。

5.3 分享标识

链接至少锁定：包 ID、条目类型、条目 ID、主版本、参数、Seed、可选绑定。示意：

/view?pkg=meshova.official@1&asset=model:chair@1&seed=7&params=...

官方条目优先短 URL。外部包可导出包文件或发布 npm 包；远程执行后期进入 Worker 沙箱，默认无 DOM、无网络、有限时与内存。

6. 全量迁移策略

不做大爆炸。先建契约和兼容适配，再按领域迁移。每个波次保持旧入口可运行；视觉与测试通过后切目录。迁移阶段不同时改变造型算法。

6.1 迁移前资产台账

先用只读扫描生成台账。每个模型/材质一行，至少记录：

字段用途
稳定 ID / 类型 / 当前入口识别重复定义、失联内容、Web 与 CLI 名称差异。
实现文件 / Web 包装 / 示例 / 测试建立完整搬迁链，不遗留第二事实源。
直接依赖与领域依赖判断 helper 留入口、进包内 shared、或保留核心。
参数 / Seed / 绑定 / 材质引用生成统一定义；保持分享与编辑能力。
当前截图 / 几何统计 / PBR 统计形成迁移前基线。
迁移状态 / 负责人 / 阻塞项控制全量重构进度。

6.2 单个模型迁移模板

  1. 锁定当前默认参数、Seed、多视角截图、部件名、三角面数范围、包围盒。
  2. 创建 *.model.ts，把 Web Schema、标签、绑定合并到定义。
  3. 把具体 build 逻辑从 src/models 移入 PCG 包。
  4. 仅该模型使用的 helper 放模型目录；同领域复用 helper 放 shared/<domain>。
  5. 通过兼容导出保持旧示例和测试短期可用。
  6. 让 Catalog 驱动 Web；删除 procmodels.js 中对应手工包装与特判。
  7. 迁移测试到 PCG 包；运行结构、确定性、截图回归。
  8. 所有消费者切换后删除旧实现与兼容导出。

6.3 单个材质迁移模板

  1. 锁定参数、Seed、PBR 通道统计、预览球截图、导出文件集合。
  2. 区分底层算子与成品配方；Buffer、Filter、PBR 校验留核心。
  3. 具体材质、批次集合、风格系统移入 pcg/official/materials。
  4. 复用领域函数移入 shared/materials；禁止因复用一次就提升核心。
  5. 让 Catalog 生成 Web 材质列表与标签；删除手工名称数组。
  6. 迁移烘焙测试与截图；验证浏览器、Node 结果一致。
  7. 所有消费者切换后删除旧 preset/批次导出。

每个现有模型/材质都有稳定 ID、入口、测试/截图、迁移波次。
P1

PCG 契约与自动 Catalog

只建最小契约：定义、校验、发现、加载。避免提前做市场、远程安装、复杂依赖解析。

交付
defineModel、defineMaterial、definePcgPackage、参数/绑定类型、构建期扫描器。
退出条件
未改 Web 注册表即可发现一个新模型和一个新材质；Node 与浏览器加载同一入口。
P2

纵向试点

迁移简单模型、复杂模型、绑定模型、基础材质、复杂材质。暴露契约缺口。

交付
M0 + T0 试点、兼容导出、Web 参数面板、通用截图与导出命令。
退出条件
试点输出达基线；新内容加入时核心与 Viewer 源码零修改。
P3

Web 注册层收敛

让 Catalog 成为模型库、材质库、搜索、分类、参数、绑定的唯一内容来源。

交付
动态加载、分享 URL、通用绑定适配、内容搜索、错误隔离。
退出条件
procmodels.js 不再新增条目；materials.js 不再维护内容名称数组。
P4

全量模型迁移

按 M1～M6 推进。每波结束再清理对应旧路径，禁止跨波次提前删除。

交付
全部模型脚本、领域 shared、包内测试与预览、旧入口迁移记录。
退出条件
模型 Catalog 覆盖现有库；src/models 无真实实现；Web 与 CLI 全部走包。
P5

全量材质迁移

按 T1～T5 推进。每次迁移明确“核心算子”和“成品配方”。

交付
全部材质脚本、包内材质 shared、通用浏览器烘焙桥、回归截图。
退出条件
具体材质不再从核心导出；Node 与浏览器同 Seed 输出一致。
P6

核心瘦身与兼容清理

最后再收紧公共导出，避免迁移中反复破坏消费者。

交付
核心入口白名单、弃用清单、包边界 lint、依赖方向检查、迁移指南。
退出条件
PCG 包可依赖核心；核心与 Web 不反向依赖官方内容；旧注册与兼容层清零。
P7

外部扩展与发布验证

核心与官方包稳定后，再验证第三方包。先本地安装，后远程沙箱。

交付
最小第三方示例包、打包命令、版本锁定、能力声明、安全执行策略。
退出条件
仓库外包能在 CLI 与 Web 预览；不修改 Meshova 核心或官方包。

8. 测试与验收

8.1 分层测试

层测试失败表示
内核单元数学、拓扑、Buffer、PBR、随机、导入导出、错误边界稳定算法或契约回归。
包契约ID 唯一、版本合法、参数默认值合法、无禁用依赖、输出类型正确内容无法被通用运行时安全加载。
确定性同版本 + 同参数 + 同 Seed 的结构哈希一致无法复现、分享、截图回归或 AI 迭代。
几何质量有限坐标、有效索引、法线、包围盒、面数范围、空部件、退化面模型迁移或运行异常。
材质质量通道尺寸、有限值、PBR 范围、法线长度、浏览器/Node 差异材质不可用或跨端不一致。
视觉回归标准视角截图 + 容差比较 + 人工抽检造型、分组、材质或相机表现变化。
Web E2E目录发现、搜索、加载、调参、绑定、分享链接、错误隔离传播与用户使用链中断。
边界测试核心不得 import pcg；Viewer 不得引用具体资产 ID分层再次腐化。

8.2 每个迁移条目的验收门

  • 默认参数输出可生成，Seed 可复现。
  • 参数最小值、最大值、典型值均不报错。
  • 部件语义标签保留；原始 importer 名称仅作内部键。
  • Web、CLI、测试使用同一入口。
  • 旧截图与新截图在约定容差内；有意变化必须单独记录。
  • 旧注册、旧包装、旧导出无剩余消费者。
  • 新实现不向核心加入内容专用函数。

9. 版本、依赖与治理

9.1 版本规则

对象规则
内核SemVer。破坏核心类型/函数必须升主版本；提供弃用周期和迁移说明。
PCG 包独立 SemVer。领域 helper 可在包内变化；对外条目 ID 与主版本决定分享链接兼容。
模型/材质条目具有内容版本。默认参数或视觉大幅变化时升条目版本，不悄悄改变旧分享结果。
生成目录无手工版本；由包版本与条目版本推导。

9.2 依赖方向

允许：PCG 模型 ─→ 包内 shared ─→ Meshova 内核
允许：PCG 材质 ─→ 包内 shared ─→ Meshova 内核
允许：Web / CLI ─→ Catalog ─→ PCG 条目 ─→ 内核

禁止：Meshova 内核 ─→ 官方 PCG 包
禁止：Web 查看器 ─→ 某个具体模型文件
禁止：模型 A ─→ 另一个模型入口的内部实现
禁止：运行时全局自注册副作用

9.3 变更审查问题

  • 这段代码描述通用计算，还是某类资产怎么生成？后者进 PCG 包。
  • 新增模型是否需要改核心或 Viewer？若需要，先判断契约缺口还是特判。
  • 共享 helper 是否已有两个领域消费者？没有则留局部。
  • 改变默认值是否会破坏旧分享链接？需要时保留旧条目版本。
  • AI 阅读入口是否需要跨越过多文件？入口应短，helper 应按语义命名。

10. 风险与控制

风险影响控制
迁移变成造型重写无法判断架构回归还是视觉改动。搬迁与美术改进分开提交；先做行为等价迁移。
“稳定内核”过瘦各 PCG 内容重复实现基础算法。允许包内 shared；通过真实复用数据再晋升核心。
“稳定内核”继续膨胀新增内容仍修改核心。建立核心晋升门槛、导出白名单、依赖边界检查。
官方 PCG 包过大Web 初始加载慢。Catalog 只带元数据；条目动态 import；按入口切块。
参数定义重复脚本与 UI 再次分叉。参数 Schema 与 build 同文件；Web 只读定义。
第三方脚本安全远程代码访问 DOM、网络或耗尽资源。外部包后置；Worker 沙箱、能力声明、限时、限内存、默认禁网。
版本链接失效传播内容不可复现。分享锁定包/条目主版本、参数、Seed；旧主版本保留可加载产物。
旧注册清理不彻底两套事实源长期共存。台账记录消费者；波次结束必须删除对应手工注册。
现有脏工作区冲突迁移覆盖并行开发。实施前划定迁移批次；小步提交；不批量格式化无关文件。

11. 完成定义

  • 所有正式模型、材质均由 PCG 包普通 TS 入口定义。
  • 核心公共入口不再导出具体模型、成品材质、内容列表。
  • 新增一个模型或材质只改 PCG 包；Catalog 自动生成。
  • web/procmodels.js 巨型手工注册被删除或缩为无内容特判的通用适配。
  • web/materials.js 不再维护多个材质名称数组。
  • Web 参数、搜索、分类、绑定、分享全部来自统一定义。
  • CLI、Agent、Web、测试使用同一条目加载链。
  • 全部迁移条目通过确定性、结构、PBR、截图、Web E2E 回归。
  • 核心对 PCG 包零反向依赖；Viewer 对具体资产 ID 零硬编码。
  • 不存在节点蓝图文件、OpPlan 内容包或脚本/图双向同步负担。
  • 仓库外最小示例 PCG 包可被安装、预览、分享，不修改核心。

Meshova 内核稳定工具箱；慢变化
PCG 包普通 TS 内容与领域能力；快变化
